# Decisions

Architecture-decision log. One entry per notable, non-obvious call — context,
decision, consequence. Ordered roughly by when each was made.

## Composition over inheritance for vehicles

**Context:** the player car (and originally traffic cars) needed a shared
drivetrain and cosmetic behavior without a rigid class hierarchy.
**Decision:** assemble vehicles from small `Node` components
(`MovementComponent`, `SwayComponent`) wired via `@export var` + `node_paths`
in the `.tscn`, rather than subclassing a base "Car" type.
**Consequence:** any component is reusable as-is on a new vehicle type with
zero code changes — but hand-edited scenes must keep `node_paths` in sync
with each script's exported vars by hand, or the reference is silently
`null` at runtime with no error. This has already caused a real bug in this
project (Day 1).

## Signals-only cross-system coupling, single wiring point

**Context:** systems (player, camera, road, HUD, gas station) need to react
to each other without becoming tangled.
**Decision:** every cross-system connection is a signal, and all of them are
connected in exactly one file, `scripts/main.gd`.
**Consequence:** any system can be swapped by re-wiring only `main.gd` /
`main.tscn` — but tracing "what happens when X occurs" always requires
starting from that one file rather than the emitting script.

## Single autoload

**Context:** run state (fuel, money, trip goal) needs to be reachable from
multiple systems without every one of them holding hard references to each
other.
**Decision:** exactly one autoload, `GameManager`, holds all of it. No other
global singletons are allowed.
**Consequence:** global state stays auditable in one file; this is a
discipline decision the team must keep enforcing as the game grows, not
something the engine prevents on its own.

## GL Compatibility renderer

**Decision:** `renderer/rendering_method = "gl_compatibility"` for both
desktop and mobile.
**Consequence:** broader hardware support and reliable headless/CI runs, at
the cost of the higher-end rendering features the Forward+ renderer offers
(not needed for flat pixel art).

## Removal of lane-switching / traffic / crash

**Context:** the Day 1 build had the player dodge lanes to avoid oncoming
traffic, with a crash ending the run. After Day 2 added fuel and money, the
resource-pressure loop alone was judged to carry more of the game's tension
than the reflex-dodge layer did, and the two didn't reinforce each other.
**Decision:** remove `LaneSwitchComponent`, `TrafficManager`, the traffic car
scene and its sprite variants, `lanes.tres`, the crash `Hitbox`, and
`RunEndReason.CRASHED` entirely. The car now only cruises; the only end
condition is `OUT_OF_FUEL`.
**Consequence — and a caught regression:** the crash `Hitbox`
(`collision_layer = 1`) had been doing double duty as the *only* thing on
collision layer 1, which the gas station's `Trigger` (`mask = 1`) depends on
to detect the player at all. A straight deletion would have silently broken
refueling — the game's other core system — with no error, only a station
that never shows its hint. Caught in self-review before commit; fixed by
adding a dedicated `PresenceArea` (layer 1, no mask, no gameplay logic of
its own) purely so the station can still see the player. See
`11_TECHNICAL_ARCHITECTURE.md` → Collision layers.

## Fixed refuel price, no income

**Decision:** refueling always costs a flat `$20` and always fills the tank
completely; no way to earn money exists.
**Consequence:** the run is finite by construction — there's no strategy
axis around income yet, only "how far can this fixed budget stretch." Income
is a planned extension point (`12_ROADMAP.md`), not yet a system.

## Full scene reload on restart

**Context:** a run needs to reset cleanly — fuel, money, road streaming
state, camera, the active gas station — all at once.
**Decision:** `restart_run()` unpauses and calls
`get_tree().reload_current_scene()` rather than resetting each system's
state in place.
**Consequence:** every pooled/streamed system guarantees a clean slate with
zero extra reset code to maintain — at the cost of a reload hitch instead of
an instant restart.

## Procedural placeholder art

**Decision:** all current sprites are generated by
`tools/generate_placeholder_art.py` rather than hand-authored.
**Consequence:** gameplay iteration isn't blocked on art production, and
regenerating art after a design change (e.g. dropping the traffic paint
jobs) is a one-command, no-asset-review operation. No real art direction has
been committed to yet.

## This documentation set replaces the original 21-file scaffold

**Context:** an earlier pass seeded 21 doc files (vision, NPCs, cities,
regions, items, story chains, save system, ...) as header-only TODO stubs,
sized for a much bigger game than what's actually been built.
**Decision:** replace that scaffold with this 7-file Bible
(`00_VISION.md` … `13_DECISIONS.md`), covering only what the codebase
actually does today.
**Consequence:** the docs describe reality instead of aspiration. Scope
beyond the current loop (NPCs, cities, story, saves) stays undocumented
until it's actually designed — see `12_ROADMAP.md` → Explicitly out of
scope right now.
